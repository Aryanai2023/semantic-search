Semantic Search

From keywords → meaning (and why it changes everything)

Embeddings Vector DB Hybrid Search RAG Evaluation

Quick framing: “Search is a product, not a feature.” Semantic search reduces ‘I can’t find it’ pain.

Why search feels broken

Users type

  • “startup doing batteries without lithium”
  • “who makes chips for edge AI cameras”
  • “paper about quantum reservoir computing finance”

But docs contain

  • synonyms (“sodium-ion”, “anode chemistry”)
  • jargon (“neuromorphic”, “on-device inference”)
  • misspellings, acronyms, different wording

Keyword search matches tokens. Semantic search matches intent.

Give one personal example: “You know it’s in the PDF somewhere but you can’t find the phrasing.”

Keyword vs Semantic (in one picture)

Keyword (lexical)

  • Exact words matter
  • Great for identifiers: SKUs, error codes
  • Fails on synonyms + paraphrases

Semantic (vector)

  • Meaning matters
  • Great for “similar to this” queries
  • Needs evaluation + guardrails
Emphasise: this is not either/or — most production systems are hybrid.

The core idea: embeddings

An embedding turns text into a numeric vector so that “similar meaning” ends up “close together”.

Close in vector space ⇒ similar concept

  • “fundraising round” ≈ “Series A”
  • “speech synthesis” ≈ “TTS”
  • “drug candidate” ≈ “lead compound”

Far apart ⇒ different concept

  • “router BGP” vs “cake recipe”
  • “invoice reconciliation” vs “gym routine”
Keep it accessible: “It’s like a GPS coordinate for meaning.”

Pipeline: end-to-end semantic search

1) Ingest PDFs, docs, notes, web pages
→
2) Chunk split into passages + metadata
3) Embed vector per chunk
→
4) Index store in vector DB
5) Query embed user question
→
6) Rank nearest neighbours + filters

Add-on: RAG uses the top passages as context to generate an answer.

Mention metadata filters: date, source, sector, geography, author, confidence.

Chunking: the underrated superpower

Bad chunking

  • too big: irrelevant noise
  • too small: missing context
  • no overlap: broken sentences

Good chunking

  • passage-sized: 150–400 tokens often works
  • overlap: keeps continuity
  • metadata: title, section, URL, date, tags
Audience takeaway: most “semantic search is bad” complaints are actually chunking + evaluation.

Vector databases (what they actually do)

  • Store: vector + text chunk + metadata
  • Search: fast “nearest neighbour” lookup (approximate kNN)
  • Filter: only search within “UK”, “FinTech”, “2024+”, “Pitch deck”
  • Return: top-k chunks + similarity scores

In practice: choose based on scale, latency, cost, and ops simplicity.

Optional examples: FAISS (local), pgvector (Postgres), managed vector DBs.

Hybrid search: best of both worlds

Lexical helps when…

  • exact names: “ASML”, “AWS”, “BGP”
  • numbers / IDs: “error 0x800”
  • rare entities + spelling matters

Semantic helps when…

  • synonyms & paraphrases
  • concept search (“similar startups”)
  • messy notes / OCR text

Production pattern: hybrid retrieval → rerank → (optional) answer.

Mention rerankers: cross-encoders / LLM reranking improves quality a lot.

RAG: “Search + Answer” (without hallucinating… too much)

  1. Retrieve top chunks
  2. Ground the model using those chunks
  3. Generate an answer with citations

Guardrails: cite sources abstain if missing quote exact lines show confidence

Use the phrase: “RAG is a reliability strategy, not a magic wand.”

Minimal Python (embeddings + FAISS)


# pip install sentence-transformers faiss-cpu
from sentence_transformers import SentenceTransformer
import faiss, numpy as np

docs = ["Sodium-ion batteries are cheaper than lithium-ion...",
        "Edge AI cameras run inference on-device to reduce latency...",
        "Quantum reservoir computing can model time series..."]

model = SentenceTransformer("all-MiniLM-L6-v2")
X = model.encode(docs, normalize_embeddings=True)

index = faiss.IndexFlatIP(X.shape[1])   # cosine similarity via inner product
index.add(np.array(X, dtype="float32"))

q = "batteries without lithium"
qv = model.encode([q], normalize_embeddings=True).astype("float32")
scores, ids = index.search(qv, k=2)

print([(docs[i], float(scores[0][j])) for j,i in enumerate(ids[0])])
Explain: normalise embeddings → inner product behaves like cosine similarity.

How do we know it’s “good”?

Offline evaluation

  • Create query → relevant docs pairs
  • Metrics: Recall@k, MRR, NDCG
  • Track quality per source + topic

Online evaluation

  • A/B test ranking changes
  • Measure: click-through, time-to-answer, “found it” rate
  • Collect “bad results” feedback loop
Key line: “If you can’t measure search quality, you can’t improve it.”

Common failure modes (and fixes)

  • Feels random → improve chunking, add reranking, add filters
  • Misses obvious results → hybrid search, better metadata, synonyms
  • Hallucinated answers → enforce citations, abstain policy, smaller context
  • Costs too much → cache embeddings, batch ingest, compress vectors
  • Privacy concerns → on-prem embeddings, encrypt at rest, access control
Useful line: “Most ‘LLM issues’ are actually retrieval issues.”

Use case example: VC / deal sourcing

Inputs

Queries
